# SPEC.md — Iklo repo specification

Repo-level spec. The stable, cross-cutting rules shared by every epic. Per-epic
specs live under [`spec/<epic>/SPEC.md`](spec/) and inherit everything here —
they only add their own **Objective** and **Success Criteria** (and override a
section when they genuinely differ).

This project uses **spec-driven development**. See [Workflow](#workflow) below.

## Objective

See [AGENTS.md](AGENTS.md) for the short pitch and [LANGUAGE.md](LANGUAGE.md)
for the full language reference.

- **Users:** language learners and tinkerers (the audience Logo classically
  served) *plus* power users who want a scriptable Unix shell that shares one
  grammar with the language, *plus* developers who want an in-process,
  transactional live-image runtime for scripting, prototyping, and agentic work.
- **Success (overall):** Iklo runs as a language, a shell, and an in-process
  database — all sharing one grammar — with the transactional live image
  backed by a real storage/VM substrate (initial candidate: Turso/VDBE, see
  [ADR-0001](design/decisions/ADR-0001-turso-vdbe-image-store.md)).

## Tech Stack

- **Language:** Rust, edition 2021, workspace `resolver = "2"`.
- **Runtime crates** (`crates/`):
  - `iklo-lexer` — tokeniser (logos-based); emits `Lexeme` values.
  - `iklo-ast`   — AST types (`Expr`, `Program`) with source spans.
  - `iklo-parser` — Pratt precedence parser; whitespace-sensitive infix operators; newline as soft terminator.
  - `iklo-runtime` — tree-walking interpreter; `Env` with transactional live-image semantics (`begin`/`commit`/`rollback`, revision counter).
  - `iklo-cli`    — `iklo` binary: file runner + multi-line REPL with `.`-prefixed commands.
- **Toolchain:** [`mise`](https://mise.jdx.dev) (`mise.toml` pins Rust and Java).
- **Build wrapper:** `Makefile` — thin cover over `cargo`.

Future crates (not yet in the workspace, expected as epics land):

- `iklo-image` — the `ImageStore` capability boundary planned in [ADR-0001](design/decisions/ADR-0001-turso-vdbe-image-store.md).
- `iklo-vdbe`  — VDBE codegen backend, once (and only if) the image store is proven.
- `iklo-shell` — shell-mode front-end sharing the parser.

## Commands

```
Build (dev):                    make build                # ≡ cargo build
Test  (dev):                    make test                 # ≡ cargo test
Release build:                  make release              # ≡ cargo build --release
Clean:                          make clean                # ≡ cargo clean
Test one crate:                 cargo test -p iklo-parser
Run the REPL:                   cargo run -p iklo-cli
Run a file:                     cargo run -p iklo-cli -- examples/hello.iklo
```

## Project Structure

```
SPEC.md                → this file (repo-level spec)
AGENTS.md              → agent operating guide (short, factual)
LANGUAGE.md            → language reference (aspirational + implemented)
README.md              → outward-facing overview
spec/                  → per-epic specs (spec-driven development)
  <epic>/SPEC.md       → one epic: objective + success criteria
tasks/                 → generated by /plan for the ACTIVE epic
  plan.md              → technical implementation plan
  todo.md              → ordered task list with acceptance criteria
crates/                → Rust runtime crates (see Tech Stack)
examples/              → runnable .iklo programs
  planned/             → .iklo programs showcasing not-yet-implemented syntax
design/decisions/      → ADRs (ADR-NNNN, never deleted; superseded/amended)
refs/                  → third-party reference material (UCBLogo, NetLogo — not code)
```

Tests currently live inline as `#[cfg(test)] mod tests` per crate. A dedicated
`tests/` layout is not planned until it's genuinely justified by the code.

## Code Style

Rust, idiomatic per crate. Kebab-case is the canonical style for Iklo-level
identifiers (primitives, keywords, forms). See
[`.github/instructions/rust.instructions.md`](.github/instructions/rust.instructions.md)
for the detailed Rust conventions we follow, and the
[self-explanatory-code-commenting](.github/instructions/self-explanatory-code-commenting.instructions.md)
instructions on when *not* to write comments.

## Testing Strategy

- **Framework:** Rust built-in `#[test]`; run via `make test` / `cargo test`.
- **Location:** inline `#[cfg(test)] mod tests` per crate.
- **Gate:** the full `cargo test` suite must pass before every commit; new
  behavior lands with a failing test first (TDD, per `/build`).

## Boundaries

- **Always:**
  - Run `make test` before committing.
  - Commit before ending any task that touched code (see [AGENTS.md § Commit rules](AGENTS.md#commit-rules)).
  - Record significant or expensive-to-reverse decisions as an ADR under `design/decisions/`.
- **Ask first:**
  - Changing the grammar or sigils.
  - Adding a new crate or dependency.
  - Breaking-change primitive redefinitions.
  - Anything touching more than ~5 files at once.
- **Never:**
  - Commit secrets.
  - Remove or skip failing tests to make a build pass.
  - Rewrite pushed history without explicit permission.
  - Introduce parse ambiguity into the reader.

## Success Criteria

Overall (per-epic criteria live in each `spec/<epic>/SPEC.md`):

- `make build`, `make test`, and `make release` all succeed.
- Every shipped feature is covered by a test and reflected in
  [LANGUAGE.md](LANGUAGE.md) and, when relevant, [AGENTS.md](AGENTS.md).
- Load-bearing decisions have a corresponding ADR.

## Workflow

Spec-driven development, four gated phases:

1. **`/spec`** — author or update a spec (this file for cross-cutting concerns; a
   `spec/<epic>/SPEC.md` for a feature/epic).
2. **`/plan`** — read the relevant spec, generate `tasks/plan.md` +
   `tasks/todo.md` for the epic you're activating. **One epic in flight at a
   time.**
3. **`/build`** — implement the next task (RED → GREEN → regression → build →
   commit). Optionally `/build auto` runs the whole plan after one approval.
4. **`/review`, `/ship`** — review the change set and open a PR.

**Bugs** are tracked as **GitHub Issues** on
[`rsenna/iklo`](https://github.com/rsenna/iklo/issues) (label `bug`), not in a
file ledger. Promote an issue into a `spec/` epic when it grows into real
design work.

### Decisions & ADRs

Three homes, by kind of content:

- **ADR** (`design/decisions/`) = *why*. Significant or expensive-to-reverse
  decisions, with context, alternatives rejected, and consequences. Sequential
  `ADR-NNNN`; statuses are Accepted / Superseded / Deprecated. **ADRs are
  never deleted** — supersede instead. A new ADR that only *narrows* an older
  one *amends* it (old ADR stays Accepted and gains an `— amended by ADR-XXXX`
  note).
- **`spec/`** = *what to build*. Objective + success criteria.
- **Inline `**Decision (date)**` notes** = lightweight, in-flight rationale.
  Promote to an ADR when the decision becomes load-bearing or gets questioned
  a second time.

Defects are none of these — they are GitHub Issues (above).

### Task lifecycle & `CLEANING_TASKS`

One epic is in flight at a time, so `tasks/plan.md` and `tasks/todo.md` are
**overwritten** when you switch epics. There is no `tasks/archive/`; completed
plans live on in git history.

Every commit that clears or replaces `tasks/` for a new epic must carry the
keyword **`CLEANING_TASKS`** in its subject:

```
chore: CLEANING_TASKS — reset tasks/ for the <next-epic> epic
```

`git log --grep=CLEANING_TASKS` then lists every task-reset boundary, so a
previous epic's plan/todo can be recovered
(`git show <commit>^:tasks/todo.md`) or rolled back.
